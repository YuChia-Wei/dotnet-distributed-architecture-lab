## 使用者輸入

```text
$ARGUMENTS
```

在繼續之前，您**必須**考慮使用者輸入（若非空）。

## 綱要

您正在更新位於 `.specify/memory/constitution.md` 的專案原則。此檔案是一個**範本**，包含方括號中的預留位置權杖（例如 `[PROJECT_NAME]`、`[PRINCIPLE_1_NAME]`）。您的工作是 (a) 收集/推導具體值，(b) 精確填寫範本，以及 (c) 將任何修正傳播到相依的產物中。

請遵循此執行流程：

1.  載入位於 `.specify/memory/constitution.md` 的現有原則範本。
    -   識別每個 `[ALL_CAPS_IDENTIFIER]` 形式的預留位置權杖。
    **重要**：使用者可能需要比範本中使用的原則更少或更多。如果指定了數量，請尊重該數量 - 遵循通用範本。您將相應地更新文件。

2.  收集/推導預留位置的值：
    -   如果使用者輸入（對話）提供了值，請使用它。
    -   否則，從現有的儲存庫上下文（README、文件、先前的原則版本（如果嵌入））推斷。
    -   對於治理日期：`RATIFICATION_DATE` 是原始採用日期（如果未知，請詢問或標記為 TODO），`LAST_AMENDED_DATE` 如果有變更則是今天，否則保留先前的日期。
    -   `CONSTITUTION_VERSION` 必須根據語意版本控制規則遞增：
        -   **MAJOR**：向後不相容的治理/原則移除或重新定義。
        -   **MINOR**：新增新原則/區段或實質性擴展的指南。
        -   **PATCH**：釐清、措辭、錯字修正、非語意性的細化。
    -   如果版本升級類型不明確，請在最終確定前提出理由。

3.  草擬更新後的原則內容：
    -   將每個預留位置替換為具體文本（不留下任何括號權杖，除非是專案選擇尚未定義的有意保留的範本欄位——明確說明任何保留的理由）。
    -   保留標題層次結構，一旦被替換，註解可以移除，除非它們仍然提供釐清性指導。
    -   確保每個「原則」區段：簡潔的名稱行、捕捉不可協商規則的段落（或項目符號列表）、如果不明顯則提供明確的理由。
    -   確保「治理」區段列出修正程序、版本控制政策和合規性審查期望。

4.  一致性傳播檢查清單（將先前的檢查清單轉換為主動驗證）：
    -   讀取 `.specify/templates/plan-template.md` 並確保任何「原則檢查」或規則與更新後的原則保持一致。
    -   讀取 `.specify/templates/spec-template.md` 以進行範圍/需求對齊——如果原則新增/移除了強制性區段或限制，則進行更新。
    -   讀取 `.specify/templates/tasks-template.md` 並確保任務分類反映了由新原則或已移除原則驅動的任務類型（例如，可觀察性、版本控制、測試紀律）。
    -   讀取 `.specify/templates/commands/*.md` 中的每個命令檔案（包括此檔案），以驗證在需要通用指南時，沒有過時的參考（僅限於代理程式特定的名稱，如 CLAUDE）殘留。
    -   讀取任何執行時期指南文件（例如，`README.md`、`docs/quickstart.md` 或代理程式特定的指南檔案（如果存在））。更新對已變更原則的參考。

5.  產生一份同步影響報告（更新後，在原則檔案頂部作為 HTML 註解前置）：
    -   版本變更：舊 → 新
    -   修改的原則列表（如果重新命名，則為舊標題 → 新標題）
    -   新增的區段
    -   移除的區段
    -   需要更新的範本（✅ 已更新 / ⚠ 待處理），附帶檔案路徑
    -   如果任何預留位置被有意延後，則提供後續的待辦事項。

6.  最終輸出前的驗證：
    -   沒有殘留的未解釋的括號權杖。
    -   版本行與報告相符。
    -   日期為 ISO 格式 YYYY-MM-DD。
    -   原則是宣告性的、可測試的，並且沒有模糊的語言（「應該」→ 在適當情況下用 MUST/SHOULD 的理由替換）。

7.  將完成的原則寫回 `.specify/memory/constitution.md`（覆寫）。

8.  向使用者輸出最終摘要，包含：
    -   新版本和升級理由。
    -   任何標記為需要手動跟進的檔案。
    -   建議的提交訊息（例如，`docs: amend constitution to vX.Y.Z (principle additions + governance update)`）。

格式與風格要求：

-   完全按照範本中的 Markdown 標題使用（不要降級/提升層級）。
-   將長理由行換行以保持可讀性（理想情況下 <100 個字元），但不要用尷尬的斷行強制執行。
-   區段之間保留一個空行。
-   避免尾隨空白。

如果使用者提供部分更新（例如，僅修訂一個原則），仍然執行驗證和版本決定步驟。

如果缺少關鍵資訊（例如，批准日期確實未知），插入 `TODO(<FIELD_NAME>): 說明` 並將其包含在同步影響報告的延後項目下。

不要建立新範本；始終在現有的 `.specify/memory/constitution.md` 檔案上操作。
